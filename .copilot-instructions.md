# BellForge - Copilot Instruction File
You are the development partner for the BellForge project. All work must follow these rules:

============================================================
SECTION 1 - GLOBAL PRINCIPLES
============================================================
1. Deterministic Output
   - Never guess. Always reason from the existing codebase.
   - Never introduce new patterns unless explicitly requested.
   - Maintain consistent naming, structure, and layout logic.

2. Test-First Bugfixing (MANDATORY)
   - When a bug is reported or detected:
       a. Write a failing test that reproduces the bug.
       b. Run through the logic and confirm the test fails for the correct reason.
       c. Fix the bug.
       d. Confirm the test now passes.
       e. Expand the test suite with:
            - edge cases
            - boundary conditions
            - regression traps
            - common failure patterns for this class of bug
   - No bugfix is complete until the expanded test suite passes.

3. Git Workflow
   - Every change must be committed with a clear message.
   - Every push increments the version manifest.
   - After pushing, verify that the Pi updater detects and applies the update.
   - Provide commands for tagging releases when appropriate.

4. Pi Verification Workflow
   - After any backend or client change:
       a. Trigger the Pi auto-update.
       b. Query the Pi's status endpoint to confirm the new version is active.
       c. Confirm the UI reflects the updated logic.
       d. If verification fails, halt and diagnose before continuing.

============================================================
SECTION 2 - UI & LAYOUT ENGINE RULES
============================================================
1. Masonry Layout Engine
   - All cards must obey masonry flow rules:
      a. No large default gaps.
      b. Cards pack densely and fill available space.
      c. Expansion of one card triggers adaptive reflow and auto-arrange of all siblings.
      d. Never allow empty whitespace to dominate the viewport.

2. Card Behavior
   - Settings Page:
       a. Cards default to collapsed (title only).
       b. Clicking expands the card.
   c. All other cards shrink and reposition using masonry auto-arrange behavior.
   - Status Page:
       a. Cards must auto-fit the viewport.
       b. No fixed heights unless explicitly required.
       c. No phantom margins or padding.

3. Motion & Transitions
   - Use smooth, deterministic transitions.
   - No jitter, no overlap, no layout thrashing.

============================================================
SECTION 3 - DEBUGGING & LOGGING
============================================================
1. Provide step-by-step debugging instructions for both backend and Pi client.
2. Generate readable, timestamped logs.
3. Suggest breakpoints and test cases.
4. When modifying layout logic, always include:
   - before/after screenshots (if requested)
   - DOM inspection notes
   - CSS cascade analysis

============================================================
SECTION 4 - OUTPUT REQUIREMENTS
============================================================
1. When asked for code, generate complete files.
2. When asked for workflows, generate step-by-step instructions.
3. When asked for improvements, refactor cleanly and explain the changes.
4. When asked for bugfixes, ALWAYS follow the test-first workflow above.
